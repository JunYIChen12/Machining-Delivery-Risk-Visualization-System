# 半成品公共池产品原型设计

基于可信报工、BOM 与工艺路线的机加工交付风险可视化 MVP。

## 核心边界

第一版只展示：

- 公共池状态
- 订单需求状态

第一版不判断：

- 库存
- 占用
- 真实可交付
- 分配决策

## 产品背景

机加工现场不是一条订单从头到尾顺着走，而是多个订单、多个成品、多个半成品规格同时在不同工序里交叉流转。

当前 MES 的入库、出库、库存、订单占用关系不准确，不能直接作为决策依据。

产品初衷不是重做 MES，而是把车间交叉生产下相对可信的半成品生产事实可视化，辅助厂长判断交付风险。

## 数据可信边界

可信数据：

- BOM / 订单需求：半成品名称、规格、需求数量
- 工艺路线：半成品对应工序路线、工序顺序
- 报工记录：工单名称、产品名称、工序名称、设备、人员、报工数量、记录状态

不可信或第一版不使用的数据：

- 实际耗时
- 良品数量 / 不良品数量
- 创建时间
- 入库 / 出库 / 当前库存

## MVP 产品定位

第一版定位为：半成品公共池可视化看板。

它基于可信 BOM、工艺路线和报工记录，按“半成品名称 + 规格”汇总理论生产能力，并展示每种半成品关联了哪些订单需求。

## 核心对象：半成品公共池

半成品公共池不是库存池，不表示仓库里真实有多少件；它是基于报工记录和工艺路线反推出来的理论生产事实池。

最小颗粒度：

```text
半成品公共池主键 = 半成品名称 + 规格
```

注意：

- 半成品名称相同但规格不同，不能合并。
- 规格相同但半成品名称不同，也不能合并。

## 能力一：公共池状态

回答：现在理论上有多少半成品生产能力？

字段：

- 半成品名称
- 规格
- 工序进度
- 理论完成量
- 在制数量
- 当前卡点
- 关联订单数
- 总需求数量
- 风险状态

理论完成量计算口径：

```text
理论完成量 = 标准工艺路线中所有必经工序报工数量的最小值
```

## 能力二：订单需求状态

回答：哪些订单需要这个半成品？

它不是占用关系，不表示该半成品已经归属于某个订单，只表示订单需求关系。

字段：

- 半成品名称
- 规格
- 订单名称
- 成品名称
- 需求数量
- 交付日期
- 需求状态

## 页面一：半成品公共池总览

展示字段：

- 半成品
- 规格
- 工序进度
- 理论完成
- 在制
- 关联订单
- 总需求
- 状态

## 页面二：半成品关联订单需求

展示字段：

- 半成品
- 规格
- 订单
- 成品名称
- 需求数量
- 交付日期

说明：该页面只展示“哪些订单需要这个半成品”，不表示这些半成品已经被这些订单占用。

## 第一版不做什么

- 真实库存
- 订单占用
- 实际可交付判断
- 自动排产
- 交付日期预测
- 设备效率
- 人员绩效
- 质量分析

## 一句话总结

本产品第一版围绕“半成品公共池”展开，基于可信 BOM、工艺路线和报工记录，按“半成品名称 + 规格”汇总每类半成品的理论完成量、工序进度和当前卡点，并展示哪些订单需要这些半成品；系统不判断库存、不判断占用、不判断真实交付，只辅助厂长看清订单交叉生产下的半成品生产事实和订单需求关系。
